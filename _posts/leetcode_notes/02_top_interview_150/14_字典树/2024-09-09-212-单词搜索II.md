## 题目

**描述：**

给定一个 `m x n` 二维字符网格 `board` 和一个单词（字符串）列表 `words`，返回所有二维网格上的单词。

单词必须按照字母顺序，通过相邻的单元格内的字母构成，其中“相邻”单元格是那些水平相邻或垂直相邻的单元格。同一个单元格内的字母在一个单词中不允许被重复使用。

**示例 1:**

![示例1图片](https://assets.leetcode.com/uploads/2020/11/07/search1.jpg)

- **输入:** `board = [["o","a","a","n"],["e","t","a","e"],["i","h","k","r"],["i","f","l","v"]]`, `words = ["oath","pea","eat","rain"]`
- **输出:** `["eat","oath"]`

**示例 2:**

![示例2图片](https://assets.leetcode.com/uploads/2020/11/07/search2.jpg)

- **输入:** `board = [["a","b"],["c","d"]]`, `words = ["abcb"]`
- **输出:** `[]`

**提示:**

- `m == board.length`
- `n == board[i].length`
- `1 <= m, n <= 12`
- `board[i][j]` 是一个小写英文字母
- `1 <= words.length <= 3 * 10^4`
- `1 <= words[i].length <= 10`
- `words[i]` 由小写英文字母组成
- `words` 中的所有字符串互不相同

## 题解## 题解

### 这题难点不在“网格搜索”，而在“单词太多”

如果这题只让我们判断一个单词能不能在网格里找到，那就是普通的回溯搜索题。

但这里不一样。  
题目给的不是一个单词，而是一整个 `words` 列表，而且数量最多可以到 `3 * 10^4`。

这就意味着，如果我们对 `words` 里的每个单词都单独跑一遍 DFS，虽然思路上能做，但会有大量重复搜索。

因为这些单词之间往往有很多公共前缀。  
比如：

```text
app
apple
apply
```

如果分开搜，它们前面的 `app` 那段路径会被反复检查很多次。

所以这题真正的关键不是“会不会 DFS”，而是：

**怎么把这些单词的公共前缀利用起来，避免重复搜索。**

### 为什么这题自然会想到 Trie

一旦题目里出现“很多单词，而且它们可能有公共前缀”，Trie 基本就是最自然的选择。

因为 Trie 本来就是专门拿来存公共前缀的。

把所有单词先插进 Trie 之后，相同的前缀只会存一份。  
这样我们在网格里做搜索时，就不是“某个具体单词能不能找到”，而是：

**当前这条路径，能不能作为某些单词的前缀继续走下去。**

这个视角一换，问题就顺了很多。

### 如果不用 Trie，会浪费在哪

假设网格里某条路径已经拼出了：

```text
oa
```

如果你是对每个单词单独搜索，那你可能会反复去判断：

- `"oath"` 能不能从这里继续
- `"oak"` 能不能从这里继续
- `"oar"` 能不能从这里继续

但如果你把这些单词先放进 Trie，那么 `"oa"` 只需要判断一次：

- Trie 里有没有 `"oa"` 这个前缀
- 如果没有，直接停
- 如果有，再继续往下找

这就是 Trie 在这题里的真正作用：

**不是帮你直接找到答案，而是帮你尽早剪枝。**

### 搜索过程到底在做什么

这题的整体流程其实可以拆成两部分：

第一部分，先把所有单词插入 Trie。  
这样我们就把“所有目标单词”整理成了一棵前缀树。

第二部分，从棋盘的每一个格子出发做 DFS。  
但这个 DFS 不再是“盲目往四周搜”，而是要同时参考 Trie：

- 当前格子的字符，如果在 Trie 当前节点下面没有对应分支，说明这条路径不可能形成任何目标单词，直接剪掉
- 如果有，就继续往下走
- 如果走到某个 Trie 节点时发现这里正好对应一个完整单词，就把它加入答案

所以这题其实是：

**在棋盘上做回溯，同时在 Trie 上同步走前缀路径。**

### 为什么这题是“回溯 + Trie”，而不是单纯 DFS

因为棋盘搜索本身仍然有普通单词搜索那一套限制：

- 每次只能走上下左右
- 同一个格子在当前路径里不能重复使用

这部分决定了它一定要用回溯。

而 Trie 负责的是另一件事：  
告诉你当前这条路径还有没有继续搜索的必要。

所以两者分工很明确：

- **回溯** 负责在棋盘上找路径
- **Trie** 负责判断这条路径是不是某些单词的合法前缀

这就是为什么这题经典写法总是这两个一起出现。

### 为什么 Trie 节点里直接存完整单词会更方便

这题有个很好用的小技巧：  
在 Trie 的单词结尾节点里，不只是打一个 `isEnd = true`，而是直接把完整单词存进去，比如：

```java
node.word = w;
```

这样做的好处是：

当我们在 DFS 过程中走到这个节点时，就说明当前路径已经拼出了一个完整单词。  
这时候可以直接把 `node.word` 加入结果，而不需要再去额外拼接路径字符串。

这会让代码干净很多。

不然的话，你要么在 DFS 过程中维护一个字符串路径，要么找到结尾后再回头重建字符串，都会更麻烦。

### DFS 具体怎么做

遍历棋盘的每一个位置 `(i, j)`，都尝试把它当作起点开始搜索。

在 DFS 过程中：

1. 先做边界判断  
   如果越界，或者这个格子已经在当前路径里用过了，就直接返回

2. 取出当前字符 `c`

3. 看 Trie 当前节点下面有没有字符 `c` 对应的子节点
   - 如果没有，说明当前路径已经不可能匹配任何单词，直接剪枝
   - 如果有，就沿着这条 Trie 分支继续往下走

4. 如果走到的新 Trie 节点里 `word != null`，说明找到了一个完整单词  
   把它加入答案

5. 把当前格子临时标记成已访问，比如改成 `'#'`

6. 向上下左右四个方向继续 DFS

7. 搜索结束后，把当前格子恢复回来

这就是标准回溯框架，只不过中间多了一层 Trie 剪枝。

### 为什么找到单词后要把 `node.word` 置空

这个细节非常关键。

因为同一个单词可能在搜索过程中从不同路径被找到多次。  
如果你每次碰到 `node.word != null` 都直接加入结果，而不做处理，就可能重复收集同一个单词。

所以经典做法是：

```java
res.add(node.word);
node.word = null;
```

这样第一次找到之后，就把这个单词从 Trie 节点上“摘掉”，后面即使再次走到这里，也不会重复加入结果。

这个去重方式比额外开一个结果集合更直接，也更省事。

### 为什么可以直接修改棋盘来标记访问

很多回溯题都会开一个 `visited` 数组。  
但这题里，直接把当前格子临时改成 `'#'` 会更方便。

原因很简单：

- 当前格子本来只会是小写字母
- `'#'` 不会和正常字符冲突
- 递归返回后再恢复原字符即可

这样既省掉了额外的 `visited` 空间，代码也更紧凑。

本质上它和 `visited` 是一回事，只是写法更简洁。

### 用示例理解一下

比如示例 1：

```text
board = [
  ['o','a','a','n'],
  ['e','t','a','e'],
  ['i','h','k','r'],
  ['i','f','l','v']
]
words = ["oath","pea","eat","rain"]
```

先把这四个单词都插进 Trie。

然后从棋盘每个格子出发搜索。

比如从左上角 `'o'` 开始：

- `'o'` 在 Trie 里是某些单词的前缀，可以继续
- 接着可以走到 `'a'`
- 再走到 `'t'`
- 再走到 `'h'`

这样就找到了 `"oath"`。

再从别的位置搜索时，也会找到 `"eat"`。

而像 `"pea"`、`"rain"`` 这种，可能在某一步就因为 Trie 里没有对应前缀而被提前剪掉，不会无谓地把所有路径都走完。

这就是 Trie 剪枝真正带来的提升。

### 代码

```java
class Solution {
    class TrieNode {
        String word;
        TrieNode[] children = new TrieNode[26];
    }

    public List<String> findWords(char[][] board, String[] words) {
        TrieNode root = new TrieNode();
        for (String w : words) {
            TrieNode node = root;
            for (char c : w.toCharArray()) {
                int idx = c - 'a';
                if (node.children[idx] == null) node.children[idx] = new TrieNode();
                node = node.children[idx];
            }
            node.word = w;
        }

        List<String> res = new ArrayList<>();
        for (int i = 0; i < board.length; i++) {
            for (int j = 0; j < board[0].length; j++) {
                dfs(board, i, j, root, res);
            }
        }
        return res;
    }

    private void dfs(char[][] board, int i, int j, TrieNode node, List<String> res) {
        if (i < 0 || j < 0 || i >= board.length || j >= board[0].length || board[i][j] == '#') {
            return;
        }

        char c = board[i][j];
        int idx = c - 'a';

        if (node.children[idx] == null) {
            return;
        }

        node = node.children[idx];

        if (node.word != null) {
            res.add(node.word);
            node.word = null;
        }

        board[i][j] = '#';

        dfs(board, i + 1, j, node, res);
        dfs(board, i - 1, j, node, res);
        dfs(board, i, j + 1, node, res);
        dfs(board, i, j - 1, node, res);

        board[i][j] = c;
    }
}
```

### 这段代码最关键的地方

这题最关键的不是回溯模板本身，而是下面这个判断：

```java
if (node.children[idx] == null) {
    return;
}
```

这句就是 Trie 剪枝的核心。

它的含义是：

**当前棋盘路径虽然还能继续走，但它已经不再是任何目标单词的前缀了，所以没必要再搜。**

也就是说，这题真正快起来，不是因为 DFS 本身优化了，而是因为 Trie 帮你把大量“不可能成功的路径”提前砍掉了。

### 为什么这题不能只靠回溯硬搜

如果没有 Trie，你只能从每个起点开始，暴力尝试很多路径，再拿结果去和 `words` 比较，或者对每个单词分别做一次搜索。

这两种方式本质上都会有大量重复。

因为很多单词共享前缀，而很多棋盘路径在前几步就已经注定不可能匹配任何目标单词。  
Trie 的价值就在于把这两类浪费一起压掉：

- 相同前缀只存一次
- 无效前缀尽早剪枝

这就是这题和普通单词搜索题最大的区别。

### 复杂度怎么理解

构建 Trie 的时间复杂度是所有单词总字符数之和，记作 `K`。

搜索阶段，理论最坏情况仍然可能很大，因为棋盘上的 DFS 本质上还是分支搜索。  
但这题真正想表达的重点不是去死磕一个特别精确的最坏复杂度式子，而是要明白：

**Trie 能大幅减少很多原本会白走的搜索路径。**

所以虽然理论上 DFS 仍然可能很深，但实际运行效率会比“逐个单词单独搜索”好得多。

### 这题真正值得记住的地方

这题最值得记住的，不是“Trie + 回溯”这六个字本身。

真正值得记住的是这个模式：

**当题目要求在一个搜索空间里匹配很多字符串，而且这些字符串有大量公共前缀时，可以先把字符串建成 Trie，再在搜索过程中同步匹配前缀。**

这是一个很常见的组合套路。

- 搜索空间可能是棋盘
- 也可能是文章、文本流、字符路径
- 只要有“很多目标串 + 公共前缀 + 逐步匹配”这三个特征，Trie 往往都很有用

这题就是这个套路里很经典的一道。

### 完整代码

```java
import java.util.ArrayList;
import java.util.List;

public class Main {

    static class Solution {

        class TrieNode {
            String word;
            TrieNode[] children = new TrieNode[26];
        }

        private TrieNode buildTrie(String[] words) {
            TrieNode root = new TrieNode();

            for (String w : words) {
                TrieNode node = root;
                for (char c : w.toCharArray()) {
                    int idx = c - 'a';
                    if (node.children[idx] == null) {
                        node.children[idx] = new TrieNode();
                    }
                    node = node.children[idx];
                }
                node.word = w;
            }

            return root;
        }

        public List<String> findWords(char[][] board, String[] words) {
            List<String> res = new ArrayList<>();
            TrieNode root = buildTrie(words);

            for (int i = 0; i < board.length; i++) {
                for (int j = 0; j < board[0].length; j++) {
                    dfs(board, i, j, root, res);
                }
            }

            return res;
        }

        private void dfs(char[][] board, int i, int j, TrieNode node, List<String> res) {
            if (i < 0 || i >= board.length || j < 0 || j >= board[0].length || board[i][j] == '#') {
                return;
            }

            char c = board[i][j];
            int idx = c - 'a';

            if (node.children[idx] == null) {
                return;
            }

            node = node.children[idx];

            if (node.word != null) {
                res.add(node.word);
                node.word = null;
            }

            board[i][j] = '#';

            dfs(board, i + 1, j, node, res);
            dfs(board, i - 1, j, node, res);
            dfs(board, i, j + 1, node, res);
            dfs(board, i, j - 1, node, res);

            board[i][j] = c;
        }
    }

    public static void main(String[] args) {
        Solution solution = new Solution();

        char[][] board1 = {
            {'o', 'a', 'a', 'n'},
            {'e', 't', 'a', 'e'},
            {'i', 'h', 'k', 'r'},
            {'i', 'f', 'l', 'v'}
        };
        String[] words1 = {"oath", "pea", "eat", "rain"};

        System.out.println("示例 1 输出: " + solution.findWords(board1, words1));

        char[][] board2 = {
            {'a', 'b'},
            {'c', 'd'}
        };
        String[] words2 = {"abcb"};

        System.out.println("示例 2 输出: " + solution.findWords(board2, words2));

        char[][] board3 = {
            {'a', 'p', 'p'},
            {'p', 'l', 'e'}
        };
        String[] words3 = {"app", "apple"};

        System.out.println("示例 3 输出: " + solution.findWords(board3, words3));
    }
}
```
