## 题目

**描述：**

给定一个长度为 `n` 的整数数组 `height` 。有 `n` 条垂线，第 `i` 条线的两个端点是 `(i, 0)` 和 `(i, height[i])` 。

找出其中的两条线，使得它们与 `x` 轴共同构成的容器可以容纳最多的水。

返回容器可以储存的最大水量。

**说明：** 你不能倾斜容器。

**示例 1:**

- **输入:** `[1,8,6,2,5,4,8,3,7]`
- **输出:** 49
- **解释:** 图中垂直线代表输入数组 `[1,8,6,2,5,4,8,3,7]`。在此情况下，容器能够容纳水（表示为蓝色部分）的最大值为 49。

**示例 2:**

- **输入:** `height = [1,1]`
- **输出:** 1

**提示:**

- `n == height.length`
- `2 <= n <= 10^5`
- `0 <= height[i] <= 10^4`

## 题解

### 先抓住这题到底在算什么

任意选两条线，能装多少水，不是由更高的那条线决定，而是由**更短的那条线**决定。

因为水面最多只能和短板齐平，所以面积其实是：

```text
面积 = 宽度 × 高度
     = (right - left) × min(height[left], height[right])
```

这里的宽度很好理解，就是两条线之间的距离；真正关键的是后面的 `min(...)`。

这也是这题最容易想偏的地方。很多人会下意识去找“最高的两条线”，但这题不是单纯比高度，还要同时看距离，而且最后真正起限制作用的还是短板。

### 暴力能做，但没有必要

最直接的做法当然是枚举所有两条线，计算它们能装多少水，然后取最大值。

这个思路没有问题，但时间复杂度是 $O(n^2)$。  
数组长度最大能到 $10^5$，这种做法肯定不够用。

所以这题真正要解决的问题是：

**能不能在不枚举所有组合的前提下，快速排除掉一大批不可能更优的情况？**

### 双指针为什么会出现在这里

既然面积同时取决于“宽度”和“短板高度”，那一个很自然的想法是：先从**最宽**的情况开始看。

也就是：

- `left` 放在最左边
- `right` 放在最右边

这样一开始两条线之间的距离最大。

接下来每次算出当前面积，再考虑下一步该怎么移动指针。

问题的关键就在这里：  
**宽度每动一次都会变小，那还值得继续往里收吗？**

答案是值得，因为虽然宽度变小了，但短板有可能变高。只要短板提高得足够多，面积还是可能变大。

### 这题真正的核心：为什么一定移动较短的那一边

假设当前两端的高度分别是：

- `height[left]`
- `height[right]`

当前面积由较短的那一边决定。

如果 `height[left] < height[right]`，那么当前短板就是左边。  
这时候如果你去移动右边，会发生什么？

- 宽度变小了
- 短板还是左边这条线，或者变得更低
- 那面积不可能变大

也就是说，**移动较高的一边没有意义**。  
因为限制面积的瓶颈根本不在它那里。

只有移动较短的那一边，才有机会遇到更高的线，把短板抬起来。虽然宽度会变小，但高度上限有可能被改善，所以面积才存在变大的可能。

这就是这题的关键结论：

**每次都移动较短的那一边。**

### 用示例走一遍

看示例 1：

```text
height = [1,8,6,2,5,4,8,3,7]
```

一开始：

- `left = 0`，高度是 `1`
- `right = 8`，高度是 `7`

当前面积：

```text
(8 - 0) × min(1, 7) = 8 × 1 = 8
```

短板是左边的 `1`，所以左指针右移。

然后变成：

- `left = 1`，高度是 `8`
- `right = 8`，高度是 `7`

当前面积：

```text
(8 - 1) × min(8, 7) = 7 × 7 = 49
```

这时已经得到了答案里的最大值 `49`。

后面继续往里收缩，虽然还会算出很多面积，但都不会超过这个值。

这道题做多了之后，你会慢慢发现，它不是在“找最优的两条线”，而是在“不断排除不可能更优的一侧”。

### 为什么这样不会漏掉答案

这题双指针能成立，不是因为“试试看也许可以”，而是因为每次移动都能排掉一整类无效情况。

比如当前左边更短，那以当前左边作为边界，配上任何更靠内的右边：

- 宽度只会更小
- 高度上限仍然受左边这条短板限制

所以这些组合都不可能比当前更好。

既然如此，当前这条较短的边就没有保留价值了，直接移动它就行。

也正因为每一步都能确定地丢掉一部分不可能成为答案的情况，所以整个过程只需要线性扫描一次。

### 代码

```java
public int maxArea(int[] height) {
    int left = 0;
    int right = height.length - 1;
    int maxArea = 0;

    while (left < right) {
        int currentArea = (right - left) * Math.min(height[left], height[right]);
        maxArea = Math.max(maxArea, currentArea);

        if (height[left] < height[right]) {
            left++;
        } else {
            right--;
        }
    }

    return maxArea;
}
```

时间复杂度是 $O(n)$，因为两个指针都只会朝中间移动一次。  
空间复杂度是 $O(1)$，只用了几个额外变量。

### 这题真正值得记住的地方

这题最值得记住的，不只是“谁矮谁移动”这个结论，而是这个结论背后的原因：

**当一个结果同时受“范围”和“瓶颈”限制时，优先去动瓶颈，才有可能变好。**

在这道题里：

- 范围是两条线之间的宽度
- 瓶颈是较短的那条线

宽度每次都会缩小，这是没法避免的；那唯一可能把面积拉回来的，就只有让短板变高。

所以移动短板，不是技巧，而是这道题本身的结构决定的。

### 完整代码

```java
import java.util.Arrays;

public class Main {

    public static int maxArea(int[] height) {
        int left = 0;
        int right = height.length - 1;
        int maxArea = 0;

        while (left < right) {
            int h = Math.min(height[left], height[right]);
            int w = right - left;
            int area = h * w;

            maxArea = Math.max(maxArea, area);

            if (height[left] < height[right]) {
                left++;
            } else {
                right--;
            }
        }

        return maxArea;
    }

    public static void main(String[] args) {
        int[] height1 = new int[]{1, 8, 6, 2, 5, 4, 8, 3, 7};

        System.out.println("示例 1 输入:");
        System.out.println("height = " + Arrays.toString(height1));

        int result1 = maxArea(height1);

        System.out.println("示例 1 输出: " + result1);
        System.out.println("-----------------------------");

        int[] height2 = new int[]{1, 1};

        System.out.println("示例 2 输入:");
        System.out.println("height = " + Arrays.toString(height2));

        int result2 = maxArea(height2);

        System.out.println("示例 2 输出: " + result2);
    }
}
```
