## 题目

**描述：**

给定一个字符串 `s` 和一个字符串数组 `words` 。`words` 中所有字符串 **长度相同** 。

`s` 中的 **串联子串** 是指一个包含 `words` 中所有字符串以任意顺序排列连接起来的子串。

- 例如，如果 `words = ["ab","cd","ef"]`， 那么 `"abcdef"`， `"abefcd"`，`"cdabef"`， `"cdefab"`，`"efabcd"`， 和 `"efcdab"` 都是串联子串。`"acdbef"` 不是串联子串，因为它不是 `words` 中任何排列的连接。

返回所有串联子串在 `s` 中的开始索引。你可以以 **任意顺序** 返回答案。

**示例 1:**

- **输入:** `s = "barfoothefoobarman"`, `words = ["foo","bar"]`
- **输出:** `[0,9]`
- **解释:**

  因为 `words.length == 2` 同时 `words[i].length == 3`，连接的子字符串的长度必须为 6。

  子串 `"barfoo"` 开始位置是 0。它是 `words` 中以 `["bar","foo"]` 顺序排列的连接。

  子串 `"foobar"` 开始位置是 9。它是 `words` 中以 `["foo","bar"]` 顺序排列的连接。

  输出顺序无关紧要。返回 `[9,0]` 也是可以的。

**示例 2:**

- **输入:** `s = "wordgoodgoodgoodbestword"`, `words = ["word","good","best","word"]`
- **输出:** `[]`

**示例 3:**

- **输入:** `s = "barfoofoobarthefoobarman"`, `words = ["bar","foo","the"]`
- **输出:** `[6,9,12]`

**提示:**

- `1 <= s.length <= 10^4`
- `1 <= words.length <= 5000`
- `1 <= words[i].length <= 30`
- `words[i]` 和 `s` 由小写英文字母组成

## 题解

### 这题难，不是难在滑动窗口三个字

这题第一眼看上去很像滑动窗口，但如果直接照着“无重复字符的最长子串”那种思路去写，通常很快就会卡住。

原因很简单：  
这题不是在字符串里找“满足某种字符条件的连续区间”，而是要找一段长度固定的子串，使它恰好能拆成若干个等长单词，并且这些单词的出现次数和 `words` 完全一致。

这里真正关键的，不是“窗口”本身，而是题目给了一个非常重要的条件：

**`words` 中所有单词的长度都相同。**

这个条件决定了我们不应该按“字符”去移动窗口，而应该按“单词长度”去移动。

### 先把题目翻译成更容易处理的形式

假设每个单词长度是 `w`，单词个数是 `m`，那么一个合法答案的总长度一定是：

```text
w * m
```

也就是说，如果某个起点是答案，那么从这个起点开始，后面这一整段长度固定的子串，必须能够刚好切成 `m` 个长度为 `w` 的小块，而且每一块都得出现在 `words` 里，次数也不能多不能少。

这样一来，这题其实就变成了：

**在字符串里找长度固定的一段，并判断它按 `w` 切块之后，得到的单词频次是否和 `words` 一样。**

如果暴力去做，就是每个起点都截一段出来，再逐块统计，能做，但会很慢。

### 为什么不能每次只移动 1 个字符

很多人第一次做这题，最容易忽略的一点就是：  
虽然原字符串是按字符组成的，但这里的判断单位其实已经不是字符了，而是长度固定的单词块。

比如 `words = ["foo", "bar"]`，每个单词长度都是 `3`。  
那你在判断 `"barfoo"` 这种子串时，合理的切法只能是：

```text
bar | foo
```

而不是从中间乱切。

所以如果已经从某个起点开始按长度 `3` 去切块了，那么窗口左右边界也应该每次移动 `3`，而不是移动 `1`。

可问题又来了：  
如果只从下标 `0` 开始，每次走 `3` 步，那下标 `1` 开始、下标 `2` 开始的情况就全漏掉了。

所以这题还有第二个关键点：

**不能只跑一个滑动窗口，而是要按不同的起始偏移量，分别跑多组窗口。**

### 为什么要从 0 到 w-1 都试一遍

假设单词长度是 `w`，那么所有可能的切块方式，其实只取决于起点模 `w` 的结果。

比如 `w = 3`，那字符串可以分成三类位置：

- 从 `0` 开始切：`0, 3, 6, 9, ...`
- 从 `1` 开始切：`1, 4, 7, 10, ...`
- 从 `2` 开始切：`2, 5, 8, 11, ...`

任何一个可能的答案，都一定落在这三种切法中的某一种里。  
所以只要把这 `w` 种起点都跑一遍，就不会漏解。

这一步想明白之后，这题的整体框架其实就已经出来了：

- 外层枚举起始偏移量 `0 ~ w - 1`
- 内层在这一组切法上做滑动窗口
- 窗口里维护当前单词出现次数
- 一旦某个单词超标，就收缩左边界
- 当窗口正好包含 `m` 个合法单词时，记录答案

### 窗口里到底维护什么

为了判断当前窗口是否合法，我们需要两个哈希表：

- `targetMap`：记录 `words` 里每个单词应出现多少次
- `windowMap`：记录当前窗口里每个单词已经出现了多少次

另外再维护三个量：

- `left`：窗口左边界
- `right`：窗口右边界
- `count`：当前窗口里已经装进来了多少个合法单词块

这里的“合法单词块”指的是：  
从字符串里截出来的这个长度为 `w` 的片段，至少得先出现在 `words` 里。

### 遇到三种情况怎么处理

每次让 `right` 往右走一个单词长度，取出一个新单词 `word`。接下来只会遇到三种情况。

#### 第一种：这个单词根本不在 `words` 里

那就不用再往下看了，因为当前窗口已经彻底断掉了。

比如 `words = ["foo", "bar"]`，你扫着扫着突然遇到一个 `"the"`，那说明无论这个窗口前面攒了什么，都不可能和目标匹配。

这时候直接：

- 清空 `windowMap`
- `count = 0`
- `left = right`

重新开始

这是最干脆的一种情况。

#### 第二种：这个单词在 `words` 里，而且数量没有超标

那就把它加入窗口，继续往右扩张。

这说明当前窗口暂时还是可能成为答案的，不需要急着动左边界。

#### 第三种：这个单词在 `words` 里，但数量超标了

这是这题最核心的地方。

比如 `words = ["foo", "bar"]`，目标里 `"foo"` 只能出现 `1` 次。  
结果你当前窗口里已经有一个 `"foo"` 了，现在又来了一个 `"foo"`，那窗口就不合法了。

这时不能整个重来，因为窗口左边那一段里可能还有价值。  
正确做法是不断移动 `left`，把左边的单词一个个移出去，直到当前这个超标单词的数量恢复正常。

也就是说，这题的收缩窗口，不是因为“长度太长了”，而是因为：

**某个单词的出现次数超过了允许范围。**

### 用示例把过程走一遍

看示例 1：

```text
s = "barfoothefoobarman"
words = ["foo", "bar"]
```

每个单词长度 `w = 3`，单词数 `m = 2`，所以目标总长度是 `6`。

目标词频表是：

```text
foo -> 1
bar -> 1
```

先看偏移量 `0` 这一组，也就是按长度 `3` 切：

```text
bar | foo | the | foo | bar | man
```

开始滑动：

- 读到 `"bar"`，在词表里，加入窗口
- 再读到 `"foo"`，也在词表里，加入窗口

这时窗口里刚好有 `2` 个单词，而且词频完全匹配，说明找到了一个答案，起点是 `0`。

继续往后：

- 下一个单词是 `"the"`，不在词表里
- 当前窗口直接作废，清空，重新开始

再往后：

- 读到 `"foo"`，加入窗口
- 读到 `"bar"`，加入窗口

这时候再次匹配成功，起点是 `9`。

所以答案就是 `[0, 9]`。

这道题难就难在这里：  
**窗口不是按字符滑动的，而是按单词块滑动的。**

只要这一点没拧过来，后面就很容易越写越乱。

### 代码

```java
public List<Integer> findSubstring(String s, String[] words) {
    List<Integer> res = new ArrayList<>();
    if (s == null || s.length() == 0 || words == null || words.length == 0) {
        return res;
    }

    int wordLen = words[0].length();
    int wordNum = words.length;
    int totalLen = wordLen * wordNum;

    if (s.length() < totalLen) {
        return res;
    }

    Map<String, Integer> targetMap = new HashMap<>();
    for (String word : words) {
        targetMap.put(word, targetMap.getOrDefault(word, 0) + 1);
    }

    for (int i = 0; i < wordLen; i++) {
        int left = i;
        int right = i;
        int count = 0;
        Map<String, Integer> windowMap = new HashMap<>();

        while (right + wordLen <= s.length()) {
            String word = s.substring(right, right + wordLen);
            right += wordLen;

            if (!targetMap.containsKey(word)) {
                windowMap.clear();
                count = 0;
                left = right;
            } else {
                windowMap.put(word, windowMap.getOrDefault(word, 0) + 1);
                count++;

                while (windowMap.get(word) > targetMap.get(word)) {
                    String leftWord = s.substring(left, left + wordLen);
                    windowMap.put(leftWord, windowMap.get(leftWord) - 1);
                    count--;
                    left += wordLen;
                }

                if (count == wordNum) {
                    res.add(left);

                    String leftWord = s.substring(left, left + wordLen);
                    windowMap.put(leftWord, windowMap.get(leftWord) - 1);
                    count--;
                    left += wordLen;
                }
            }
        }
    }

    return res;
}
```

### 代码里最不好理解的两处

这题代码不短，但真正难理解的地方主要就两个。

#### 第一处：为什么找到答案后，还要继续把左边弹掉一个单词

这段代码很多人第一次看会觉得有点突兀：

```java
String leftWord = s.substring(left, left + wordLen);
windowMap.put(leftWord, windowMap.get(leftWord) - 1);
count--;
left += wordLen;
```

原因是这样的：

当 `count == wordNum` 时，说明当前窗口已经正好匹配成功了。  
但题目要求的是找出**所有**起点，所以不能停下来，还得继续向后找。

而窗口接下来如果还要向前滑动，最自然的做法就是先把最左边那个单词移出去，这样后面新进来的单词才有机会和剩余部分重新组成新的答案。

这一步不是“修错”，而是为了继续搜索下一个可能的匹配位置。

#### 第二处：为什么收缩窗口时只看当前超标单词

因为窗口原本一直都是合法推进的。  
一旦不合法，问题只可能出在刚刚新加入的这个单词上。

所以收缩时，条件只需要写成：

```java
while (windowMap.get(word) > targetMap.get(word))
```

意思是：只要这个新加入的单词次数超了，就一直从左边删，删到它恢复正常为止。

这是这题窗口收缩的核心逻辑。

### 复杂度怎么理解

设字符串长度为 `n`，单词长度为 `w`。

外层有 `w` 轮偏移量枚举；  
每一轮里，`left` 和 `right` 都是按步长 `w` 向前走，而且都不会回退。

所以虽然看起来是双层结构，但整体并不会退化成暴力枚举。

通常可以把时间复杂度理解为 `O(n * w)`，也常见写法会把它视作按常数个偏移量分组扫描，整体接近线性遍历。  
空间复杂度主要来自两个哈希表，用来存目标词频和窗口词频。

### 这题真正值得记住的地方

这题最值得记住的，不是“多起点滑动窗口”这个名字，而是这种拆题方式：

**一旦题目里出现“固定长度块”“任意顺序拼接”“频次匹配”这几个特征，就别再按字符级别去想，应该先把问题切换到块级别。**

本题里，正因为所有单词长度都相同，我们才有资格按单词块去滑动窗口；  
也正因为允许任意顺序，所以才需要哈希表去做频次匹配，而不是直接比较字符串。

以后再遇到这类“固定块 + 词频统计 + 连续区间”的题，基本都可以往这个方向靠。

### 完整代码

```java
import java.util.ArrayList;
import java.util.HashMap;
import java.util.List;
import java.util.Map;

public class Main {

    public static List<Integer> findSubstring(String s, String[] words) {
        List<Integer> res = new ArrayList<>();

        if (s == null || s.length() == 0 || words == null || words.length == 0) {
            return res;
        }

        int wordLen = words[0].length();
        int wordNum = words.length;
        int totalLen = wordLen * wordNum;

        if (s.length() < totalLen) {
            return res;
        }

        Map<String, Integer> targetMap = new HashMap<>();
        for (String word : words) {
            targetMap.put(word, targetMap.getOrDefault(word, 0) + 1);
        }

        for (int i = 0; i < wordLen; i++) {
            int left = i;
            int right = i;
            int count = 0;
            Map<String, Integer> windowMap = new HashMap<>();

            while (right + wordLen <= s.length()) {
                String word = s.substring(right, right + wordLen);
                right += wordLen;

                if (!targetMap.containsKey(word)) {
                    windowMap.clear();
                    count = 0;
                    left = right;
                } else {
                    windowMap.put(word, windowMap.getOrDefault(word, 0) + 1);
                    count++;

                    while (windowMap.get(word) > targetMap.get(word)) {
                        String leftWord = s.substring(left, left + wordLen);
                        windowMap.put(leftWord, windowMap.get(leftWord) - 1);
                        count--;
                        left += wordLen;
                    }

                    if (count == wordNum) {
                        res.add(left);

                        String leftWord = s.substring(left, left + wordLen);
                        windowMap.put(leftWord, windowMap.get(leftWord) - 1);
                        count--;
                        left += wordLen;
                    }
                }
            }
        }

        return res;
    }

    public static void main(String[] args) {
        String s1 = "barfoothefoobarman";
        String[] words1 = {"foo", "bar"};

        System.out.println("示例 1 输入: s = \"" + s1 + "\", words = [\"foo\", \"bar\"]");
        System.out.println("示例 1 输出: " + findSubstring(s1, words1));
        System.out.println("-----------------------------");

        String s2 = "wordgoodgoodgoodbestword";
        String[] words2 = {"word", "good", "best", "word"};

        System.out.println("示例 2 输入: s = \"" + s2 + "\", words = [\"word\", \"good\", \"best\", \"word\"]");
        System.out.println("示例 2 输出: " + findSubstring(s2, words2));
        System.out.println("-----------------------------");

        String s3 = "barfoofoobarthefoobarman";
        String[] words3 = {"bar", "foo", "the"};

        System.out.println("示例 3 输入: s = \"" + s3 + "\", words = [\"bar\", \"foo\", \"the\"]");
        System.out.println("示例 3 输出: " + findSubstring(s3, words3));
    }
}
```
