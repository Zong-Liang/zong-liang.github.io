## 题目

**描述：**

给定一个整数数组 `nums` 和一个整数目标值 `target`，请你在该数组中找出 **和为目标值 `target`** 的那 **两个** 整数，并返回它们的数组下标。

你可以假设每种输入只会对应一个答案。但是，数组中同一个元素在答案里不能重复出现。

你可以按任意顺序返回答案。

**示例 1:**

- **输入:** `nums = [2,7,11,15]`, `target = 9`
- **输出:** `[0,1]`
- **解释:** 因为 `nums[0] + nums[1] == 9` ，返回 `[0, 1]` 。

**示例 2:**

- **输入:** `nums = [3,2,4]`, `target = 6`
- **输出:** `[1,2]`

**示例 3:**

- **输入:** `nums = [3,3]`, `target = 6`
- **输出:** `[0,1]`

**提示:**

- `2 <= nums.length <= 10^4`
- `-10^9 <= nums[i] <= 10^9`
- `-10^9 <= target <= 10^9`
- **只会存在一个有效答案**

**进阶：** 你可以想出一个时间复杂度小于 `O(n^2)` 的算法吗？

## 题解

### 先看这题真正要找的是什么

这题不是让我们找“两个数”，而是让我们找：

**哪两个位置上的数，加起来刚好等于 `target`。**

这里有两个细节很重要：

- 返回的是**下标**，不是数值本身
- 同一个元素不能重复使用

所以题目真正要做的事，其实是：

**对每个数，快速判断它需要搭配的另一个数是否已经出现过。**

这也是这题最核心的切入点。

### 最直接的想法，为什么不够好

最容易想到的办法当然是枚举。

先固定第一个数 `nums[i]`，再去后面找第二个数 `nums[j]`，看它们两个相加是不是 `target`。

写出来就是两层循环：

- 外层枚举 `i`
- 内层枚举 `j > i`
- 如果 `nums[i] + nums[j] == target`，就返回 `[i, j]`

这个思路没有任何问题，而且非常直观。  
它的好处是：读起来最容易，逻辑也最直接。

但问题也很明显。  
如果数组很长，那很多配对其实都是白查的，最坏情况下几乎要把所有两两组合都试一遍，时间复杂度是 `O(n^2)`。

所以这题真正值得学的，不是暴力怎么写，而是怎么把“找另一个数”这件事做快。

### 关键转折：不要去找另一个数，改成问“我缺谁”

假设当前遍历到 `nums[i]`，那它想和谁配对，答案其实是固定的：

```text
target - nums[i]
```

这个值通常叫补数，也就是 `complement`。

比如：

- 当前数是 `2`
- 目标值是 `9`
- 那它需要的另一个数就是 `7`

所以这题可以换个角度来想：

**每看到一个数，不要再往后慢慢找它的搭档，而是直接问：我需要的那个补数，之前出现过没有？**

如果之前出现过，那答案立刻就出来了。  
如果之前没出现过，那就把当前这个数记下来，留给后面的元素来找。

这就是哈希表解法的本质。

### 为什么哈希表正适合干这件事

哈希表最擅长的，就是“快速判断某个值在不在”。

我们可以维护一个映射关系：

```text
数值 -> 下标
```

遍历数组时，对于当前元素 `nums[i]`：

1. 先算出它需要的补数 `target - nums[i]`
2. 看这个补数是否已经在哈希表里
3. 如果在，直接返回补数的下标和当前下标
4. 如果不在，就把当前数和它的下标放进哈希表

这样整个数组只需要扫一遍，效率就从 `O(n^2)` 降到了 `O(n)`。

### 这里最关键的一点：先查，再存

这题里有一个很容易被忽略、但非常关键的细节：

**必须先查补数，再把当前数字存进哈希表。**

为什么？

因为题目不允许同一个元素重复使用。  
如果你先把当前数字存进去，再去查补数，就有可能把自己和自己配对上。

看一个最典型的例子：

```text
nums = [3,3]
target = 6
```

正确答案是 `[0,1]`。  
处理第一个 `3` 时，哈希表里还没有任何元素，所以不能直接组成答案，只能先记下来。  
等处理第二个 `3` 时，发现它需要的补数也是 `3`，而这个 `3` 已经在哈希表里了，于是返回 `[0,1]`。

这个过程之所以正确，就是因为我们做的是：

- 先查
- 后存

这样才能保证配对到的是“之前出现过的元素”，而不是当前元素自己。

### 用示例走一遍过程

看示例 1：

```text
nums = [2,7,11,15]
target = 9
```

一开始哈希表为空。

#### 第 0 位：`2`

当前数是 `2`，它需要的补数是：

```text
9 - 2 = 7
```

哈希表里还没有 `7`，说明当前还配不成。  
那就把 `2` 存进去：

```text
2 -> 0
```

#### 第 1 位：`7`

当前数是 `7`，它需要的补数是：

```text
9 - 7 = 2
```

这次哈希表里已经有 `2` 了，而且它的下标是 `0`。  
说明答案已经找到了，直接返回：

```text
[0, 1]
```

整个过程到这里就结束了，不需要再往后看。

这个解法最顺的地方就在这里：

**每个数只负责两件事：先问自己缺谁，再决定要不要把自己留下。**

### 先保留一个最直观的写法

虽然这题真正推荐的是哈希表，但暴力写法还是值得先看一下。  
因为它能把题目最原始的结构讲清楚，也方便看出优化到底优化在了哪里。

```java
public int[] twoSum(int[] nums, int target) {
    for (int i = 0; i < nums.length; i++) {
        for (int j = i + 1; j < nums.length; j++) {
            if (nums[i] + nums[j] == target) {
                return new int[]{i, j};
            }
        }
    }
    return new int[0];
}
```

这个写法的问题，不是错，而是慢。  
它每次都要真的去后面找另一个数，所以时间复杂度是 `O(n^2)`。

### 真正推荐掌握的写法

```java
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();

    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];

        if (map.containsKey(complement)) {
            return new int[]{map.get(complement), i};
        }

        map.put(nums[i], i);
    }

    return new int[0];
}
```

### 这段代码到底在做什么

整段代码其实只有一个核心动作：

**把“向后找另一个数”改成“向前查补数”。**

逐行看会更清楚。

先开一个哈希表：

```java
Map<Integer, Integer> map = new HashMap<>();
```

它的作用是记录之前遍历过的数字，以及它们的下标。

然后进入循环：

```java
for (int i = 0; i < nums.length; i++)
```

每次处理一个新元素。

接着计算补数：

```java
int complement = target - nums[i];
```

这一步是在问：  
当前这个数如果要凑成 `target`，还差谁？

然后查哈希表：

```java
if (map.containsKey(complement)) {
    return new int[]{map.get(complement), i};
}
```

如果补数已经出现过，那答案就直接出来了。

如果没出现过，就把当前值存进去：

```java
map.put(nums[i], i);
```

表示以后如果有人需要这个数，就可以从哈希表里找到我。

### 这题最容易写错的地方

这题不难，但有几个细节非常容易在第一次写时出问题。

#### 第一个误区：先存再查

前面已经提过，这样容易把当前元素和自己配对，尤其是在一些特殊输入里会出错。

稳妥的顺序一定是：

- 先查补数
- 后存当前值

#### 第二个误区：把数值当答案返回

题目要的是下标，不是数字本身。  
所以哈希表里存的不是“数值出现过没有”这么简单，而是：

```text
数值 -> 下标
```

因为最后要返回的是位置。

#### 第三个误区：觉得数组排序后用双指针会更好

如果题目只是问“有没有两个数之和等于 target”，排序后双指针当然是一个思路。  
但这题要求返回原下标，一旦排序，下标顺序就被打乱了，还得额外想办法保留原位置，反而更麻烦。

所以这题最自然的主解法，就是哈希表。

### 复杂度为什么是 O(n)

暴力写法是两层循环，所以时间复杂度是 `O(n^2)`。

哈希表写法不同。  
它只遍历了一次数组，而每次循环里做的事情只有：

- 计算补数
- 查一次哈希表
- 存一次哈希表

这些操作平均都可以看成 `O(1)`，所以总时间复杂度就是：

- 时间复杂度：`O(n)`

空间方面，哈希表最坏情况下可能把前面的元素都存进去，因此：

- 空间复杂度：`O(n)`

这就是典型的“空间换时间”。

### 这题真正值得记住的地方

这题真正值得记住的，不是“两数之和”这道题本身，而是这种非常常见的处理方式：

**当题目要求你在遍历过程中快速判断“某个目标值是否已经出现过”时，优先考虑哈希表。**

本题里，这个目标值就是补数。  
你不是在找任意一个数，而是在找“能和当前数凑成目标值的那个数”。

把这个思路记住，后面再遇到补数、差值、配对、频次这类题，都会很自然地想到哈希表。

### 完整代码

```java
import java.util.Arrays;
import java.util.HashMap;
import java.util.Map;

public class Main {

    public static int[] twoSum(int[] nums, int target) {
        Map<Integer, Integer> map = new HashMap<>();

        for (int i = 0; i < nums.length; i++) {
            int complement = target - nums[i];

            if (map.containsKey(complement)) {
                return new int[]{map.get(complement), i};
            }

            map.put(nums[i], i);
        }

        return new int[0];
    }

    public static void main(String[] args) {
        int[] nums1 = new int[]{2, 7, 11, 15};
        int target1 = 9;
        System.out.println("示例 1 输入: nums = " + Arrays.toString(nums1) + ", target = " + target1);
        System.out.println("示例 1 输出: " + Arrays.toString(twoSum(nums1, target1)));
        System.out.println("-----------------------------");

        int[] nums2 = new int[]{3, 2, 4};
        int target2 = 6;
        System.out.println("示例 2 输入: nums = " + Arrays.toString(nums2) + ", target = " + target2);
        System.out.println("示例 2 输出: " + Arrays.toString(twoSum(nums2, target2)));
        System.out.println("-----------------------------");

        int[] nums3 = new int[]{3, 3};
        int target3 = 6;
        System.out.println("示例 3 输入: nums = " + Arrays.toString(nums3) + ", target = " + target3);
        System.out.println("示例 3 输出: " + Arrays.toString(twoSum(nums3, target3)));
    }
}
```
