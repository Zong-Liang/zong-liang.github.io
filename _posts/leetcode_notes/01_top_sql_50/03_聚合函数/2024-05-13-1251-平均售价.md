## 题目：1251. 平均售价

**表:** `Prices`

| Column Name | Type |
| ----------- | ---- |
| product_id  | int  |
| start_date  | date |
| end_date    | date |
| price       | int  |

- `(product_id, start_date, end_date)` 是 `prices` 表的主键。
- `prices` 表的每一行表示的是某个产品在一段时期内的价格。
- 每个产品的对应时间段是不会重叠的，这也意味着同一个产品的价格时段不会出现交叉。

**表:** `UnitsSold`

| Column Name   | Type |
| ------------- | ---- |
| product_id    | int  |
| purchase_date | date |
| units         | int  |

- 该表可能包含重复数据。
- 该表的每一行表示的是每种产品的出售日期，单位和产品 id。

**问题描述:**

编写解决方案以查找每种产品的平均售价。`average_price` 应该 **四舍五入到小数点后两位**。如果产品没有任何售出，则假设其平均售价为 0。

返回结果表 **无顺序要求**。

**示例 1:**

**输入:**

`Prices` table:

| product_id | start_date | end_date   | price |
| ---------- | ---------- | ---------- | ----- |
| 1          | 2019-02-17 | 2019-02-28 | 5     |
| 1          | 2019-03-01 | 2019-03-22 | 20    |
| 2          | 2019-02-01 | 2019-02-20 | 15    |
| 2          | 2019-02-21 | 2019-03-31 | 30    |

`UnitsSold` table:

| product_id | purchase_date | units |
| ---------- | ------------- | ----- |
| 1          | 2019-02-25    | 100   |
| 1          | 2019-03-01    | 15    |
| 2          | 2019-02-10    | 200   |
| 2          | 2019-03-22    | 30    |

**输出:**

| product_id | average_price |
| ---------- | ------------- |
| 1          | 6.96          |
| 2          | 16.96         |

**解释:**

平均售价 = 产品总价 / 销售的产品数量。

- 产品 1 的平均售价 = `((100 * 5) + (15 * 20)) / (100 + 15)` = 6.96
- 产品 2 的平均售价 = `((200 * 15) + (30 * 30)) / (200 + 30)` = 16.96

## 题解

这题不能直接对 `price` 求平均，因为题目要的是**平均售价**，本质上算的是：

```sql
总销售额 / 总销量
```

也就是说，某个产品卖得多的那一段价格，权重也会更大，所以这里求的是**加权平均价**，不是简单平均。

那这题的关键就在于：  
**每一笔销售记录，要先找到它发生当天对应的价格。**

因为 `Prices` 表里记录的是某个产品在一段时间内的价格，而 `UnitsSold` 表里记录的是某一天卖了多少件，所以必须先把两张表按下面两个条件对应起来：

- 是同一个 `product_id`；
- `purchase_date` 落在这段价格区间内。

对应的 SQL 写法如下：

```sql
SELECT p.product_id,
       IFNULL(ROUND(SUM(p.price * u.units) / SUM(u.units), 2), 0) AS average_price
FROM Prices p
LEFT JOIN UnitsSold u
       ON p.product_id = u.product_id
      AND u.purchase_date BETWEEN p.start_date AND p.end_date
GROUP BY p.product_id;
```

这条 SQL 可以按顺序这样理解。

先看连接部分：

```sql
FROM Prices p
LEFT JOIN UnitsSold u
       ON p.product_id = u.product_id
      AND u.purchase_date BETWEEN p.start_date AND p.end_date
```

这里做了两件事：

第一，先根据 `product_id` 找到同一个产品的销售记录。  
第二，再用：

```sql
u.purchase_date BETWEEN p.start_date AND p.end_date
```

确保这笔销售发生在这段价格生效期间内。

这样每一笔销量，就都能匹配到当时真正对应的单价。

接着看总销售额：

```sql
SUM(p.price * u.units)
```

它表示把每一笔销售的：

```sql
单价 × 销量
```

先算出来，再全部加总，得到这个产品的总销售额。

然后看总销量：

```sql
SUM(u.units)
```

就是这个产品一共卖出了多少件。

因此：

```sql
SUM(p.price * u.units) / SUM(u.units)
```

就是题目要求的平均售价。

再用：

```sql
ROUND(..., 2)
```

把结果四舍五入保留两位小数。

这里还有一个容易忽略的点：题目说如果某个产品**没有任何售出**，平均售价要记为 `0`。  
所以这里用了：

```sql
IFNULL(..., 0)
```

因为没有销售记录时，聚合结果会是 `NULL`，这时候就要手动补成 `0`。

之所以这里要用 `LEFT JOIN`，也是为了保证：  
即使某个产品在 `UnitsSold` 里没有对应销售记录，它也仍然会出现在结果中。

结合示例来看：

对于产品 `1`：

- `2019-02-25` 卖了 `100` 件，当时价格是 `5`；
- `2019-03-01` 卖了 `15` 件，当时价格是 `20`。

所以平均售价是：

```sql
((100 * 5) + (15 * 20)) / (100 + 15) = 6.96
```

产品 `2` 也是同样的计算逻辑。

这题真正要抓住的是两点：

- 求“平均售价”时，不是简单对价格取平均，而是算 `总销售额 / 总销量`；
- 价格是按时间段生效的，所以必须先根据销售日期，把每笔销量匹配到对应时段的价格。

以后只要看到“单价会变、销量不同、要求平均售价”这类题，就要优先想到这是**加权平均**问题。
