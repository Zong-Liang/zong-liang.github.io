## 题目：1934. 确认率

**表:** `Signups`

| Column Name | Type     |
| ----------- | -------- |
| user_id     | int      |
| time_stamp  | datetime |

- `User_id` 是该表的主键。
- 每一行都包含 ID 为 `user_id` 的用户的注册时间信息。

**表:** `Confirmations`

| Column Name | Type                         |
| ----------- | ---------------------------- |
| user_id     | int                          |
| time_stamp  | datetime                     |
| action      | ENUM('confirmed', 'timeout') |

- `(user_id, time_stamp)` 是该表的主键。
- `user_id` 是一个引用到注册表的外键。
- 该表的每一行都表示 ID 为 `user_id` 的用户在 `time_stamp` 请求了一条确认消息，该确认消息要么被确认 ('confirmed')，要么被过期 ('timeout')。

**问题描述:**

用户的 **确认率** 是 `'confirmed'` 消息的数量除以请求的确认消息的总数。没有请求任何确认消息的用户的确认率为 **0**。确认率**四舍五入到小数点后两位**。

编写一个 SQL 查询来查找每个用户的确认率。

以 **任意顺序** 返回结果表。

**示例 1:**

**输入:**

`Signups` 表:

| user_id | time_stamp          |
| ------- | ------------------- |
| 3       | 2020-03-21 10:16:13 |
| 7       | 2020-01-04 13:57:59 |
| 2       | 2020-07-29 23:09:44 |
| 6       | 2020-12-09 10:39:37 |

`Confirmations` 表:

| user_id | time_stamp          | action    |
| ------- | ------------------- | --------- |
| 3       | 2021-01-06 03:30:46 | timeout   |
| 3       | 2021-07-14 14:00:00 | timeout   |
| 7       | 2021-06-12 11:57:29 | confirmed |
| 7       | 2021-06-13 12:58:28 | confirmed |
| 7       | 2021-06-14 13:59:27 | confirmed |
| 2       | 2021-01-22 00:00:00 | confirmed |
| 2       | 2021-02-28 23:59:59 | timeout   |

**输出:**

| user_id | confirmation_rate |
| ------- | ----------------- |
| 6       | 0.00              |
| 3       | 0.00              |
| 7       | 1.00              |
| 2       | 0.50              |

**解释:**

- 用户 6 没有请求任何确认消息。确认率为 0。
- 用户 3 进行了 2 次请求, 都超时了。确认率为 0。
- 用户 7 提出了 3 个请求, 所有请求都得到了确认。确认率为 1。
- 用户 2 做了 2 个请求, 其中一个被确认, 另一个超时。确认率为 1 / 2 = 0.5。

## 题解

这题的关键有两个：

- 要为**每个注册用户**计算确认率；
- **没有任何确认记录的用户，确认率也要算成 `0`**。

所以这题不能直接从 `Confirmations` 表出发，因为那样会把没有请求过确认消息的用户漏掉。  
正确做法应该是以 `Signups` 表为主表，再去关联 `Confirmations` 表，这样所有注册用户都会先被保留下来。

这也是为什么这里要用 `LEFT JOIN`。

确认率的定义题目已经给出来了：

```sql
confirmed 的数量 / 确认请求总数
```

也就是说，分子是 `action = 'confirmed'` 的次数，分母是这个用户在 `Confirmations` 表里的总记录数。

对应的 SQL 写法如下：

```sql
SELECT s.user_id,
       ROUND(
           IFNULL(
               SUM(CASE
                       WHEN c.action = 'confirmed' THEN 1
                       ELSE 0
                   END) / COUNT(c.user_id),
               0
           ),
           2
       ) AS confirmation_rate
FROM Signups s
LEFT JOIN Confirmations c
       ON s.user_id = c.user_id
GROUP BY s.user_id;
```

这条 SQL 可以按顺序这样理解。

先看连接部分：

```sql
FROM Signups s
LEFT JOIN Confirmations c
       ON s.user_id = c.user_id
```

这里表示先保留所有注册用户，再根据 `user_id` 去找他们对应的确认记录。  
如果某个用户没有任何确认记录，那么连接后 `c` 表的字段都会是 `NULL`，但这个用户依然会出现在结果里。

然后看分子：

```sql
SUM(CASE
        WHEN c.action = 'confirmed' THEN 1
        ELSE 0
    END)
```

这一步是在统计每个用户一共有多少条 `confirmed` 记录。  
遇到 `confirmed` 就记 `1`，否则记 `0`，最后加总起来，就是确认成功的次数。

再看分母：

```sql
COUNT(c.user_id)
```

这里统计的是每个用户一共有多少条确认请求记录。  
因为 `COUNT(列名)` 只统计非 `NULL` 的值，所以对于没有确认记录的用户，这里会得到 `0`。

接下来把分子除以分母，就得到了确认率。  
但这里还有一个细节：如果某个用户一条确认记录都没有，那么分母就是 `0`，此时结果会是 `NULL`。题目要求这种用户的确认率是 `0`，所以要再用：

```sql
IFNULL(..., 0)
```

把这个 `NULL` 补成 `0`。

最后再用：

```sql
ROUND(..., 2)
```

把结果四舍五入保留两位小数。

结合示例来看：

- 用户 `6` 没有任何确认记录，所以确认率是 `0.00`；
- 用户 `3` 有两条记录，但都是 `timeout`，所以确认率是 `0.00`；
- 用户 `7` 有三条记录，而且全是 `confirmed`，所以确认率是 `1.00`；
- 用户 `2` 一共两条记录，其中一条 `confirmed`，一条 `timeout`，所以确认率是 `0.50`。

这题本质上考的是两个点：

- 需要保留主表全部数据时，要想到用 `LEFT JOIN`；
- 需要按条件统计某一类记录占总记录的比例时，要想到用 `CASE WHEN` 配合聚合函数来做条件统计。

这种“分组后算比例”的题，后面会经常遇到。
