## 题目：1321. 餐馆营业额变化增长

**表:** `Customer`

| Column Name | Type    |
| ----------- | ------- |
| customer_id | int     |
| name        | varchar |
| visited_on  | date    |
| amount      | int     |

- 在 SQL 中, `(customer_id, visited_on)` 是该表的主键。
- 该表包含一家餐馆的顾客交易数据。
- `visited_on` 表示 `(customer_id)` 的顾客在 `visited_on` 那天访问了餐馆。
- `amount` 是一个顾客某一天的消费总额。

**问题描述:**

你是餐馆的老板，现在你想分析一下可能的营业额变化增长（每天至少有一位顾客）。

计算以 7 天（某日期 + 该日期前的 6 天）为一个时间段的顾客消费平均值。`average_amount` 要 **保留两位小数**。

结果按 `visited_on` **升序排序**。

**示例 1:**

**输入:**

`Customer` 表:

| customer_id | name    | visited_on | amount |
| ----------- | ------- | ---------- | ------ |
| 1           | Jhon    | 2019-01-01 | 100    |
| 2           | Daniel  | 2019-01-02 | 110    |
| 3           | Jade    | 2019-01-03 | 120    |
| 4           | Khaled  | 2019-01-04 | 130    |
| 5           | Winston | 2019-01-05 | 110    |
| 6           | Elvis   | 2019-01-06 | 140    |
| 7           | Anna    | 2019-01-07 | 150    |
| 8           | Maria   | 2019-01-08 | 80     |
| 9           | Jaze    | 2019-01-09 | 110    |
| 1           | Jhon    | 2019-01-10 | 130    |
| 3           | Jade    | 2019-01-10 | 150    |

**输出:**

| visited_on | amount | average_amount |
| ---------- | ------ | -------------- |
| 2019-01-07 | 860    | 122.86         |
| 2019-01-08 | 840    | 120.00         |
| 2019-01-09 | 840    | 120.00         |
| 2019-01-10 | 1000   | 142.86         |

**解释:**

- 第一个七天消费平均值从 2019-01-01 到 2019-01-07 是 (100 + 110 + 120 + 130 + 110 + 140 + 150)/7 = 122.86
- 第二个七天消费平均值从 2019-01-02 到 2019-01-08 是 (110 + 120 + 130 + 110 + 140 + 150 + 80)/7 = 120
- ... 以此类推

## 题解

这题表面上是在求“7 天平均营业额”，但真正要先想明白的是：

**题目要的不是单个顾客的消费，而是每天的总营业额，再在这个基础上做 7 天滑动统计。**

因为同一天可能会有多位顾客消费，所以如果直接对原表做窗口计算，就会把同一天的多条记录拆开，结果一定不对。  
因此这题必须分两步走：

1. 先按日期把每天的营业额汇总出来；
2. 再对这些“按天汇总后的结果”做连续 7 天的滑动求和和平均值。

对应的 SQL 写法如下：

```sql
WITH DailyTotals AS (
    SELECT visited_on,
           SUM(amount) AS daily_amount
    FROM Customer
    GROUP BY visited_on
),
MovingWindow AS (
    SELECT visited_on,
           SUM(daily_amount) OVER (
               ORDER BY visited_on
               ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
           ) AS amount,
           COUNT(*) OVER (
               ORDER BY visited_on
               ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
           ) AS day_count
    FROM DailyTotals
)
SELECT visited_on,
       amount,
       ROUND(amount / 7, 2) AS average_amount
FROM MovingWindow
WHERE day_count = 7
ORDER BY visited_on;
```

这段查询可以顺着思路拆成三步理解。

第一步，先把每天的营业额合并出来：

```sql
SELECT visited_on,
       SUM(amount) AS daily_amount
FROM Customer
GROUP BY visited_on
```

这里的作用很关键。  
因为原表里一行只表示“某个顾客某天的消费”，但题目要分析的是“餐馆每天的营业额变化”。  
所以必须先按 `visited_on` 分组，把同一天所有顾客的消费加起来，得到每天的总金额。

比如示例里 `2019-01-10` 这一天有两条记录：

- Jhon 消费 `130`
- Jade 消费 `150`

那么这一天真正的营业额应该是：

- `130 + 150 = 280`

如果不先做这一步，后面的 7 天统计就会错。

第二步，在按天汇总后的结果上做滑动窗口计算：

```sql
SUM(daily_amount) OVER (
    ORDER BY visited_on
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
) AS amount
```

这部分表示：  
以当前日期为结尾，向前取 6 天，再加上当天，一共 7 天，把这 7 天的营业额加起来。

也就是说：

- 到 `2019-01-07` 时，统计的是 `2019-01-01 ~ 2019-01-07`
- 到 `2019-01-08` 时，统计的是 `2019-01-02 ~ 2019-01-08`
- 依次往后滑动

这就是典型的 7 天滑动窗口。

同时还要配合这一句：

```sql
COUNT(*) OVER (
    ORDER BY visited_on
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
) AS day_count
```

它用来计算当前窗口里一共有几天的数据。

为什么要多算这个值？  
因为在最开始的几天里，窗口还不满 7 天。  
比如第一天只有 1 天数据，第二天只有 2 天数据，直到第 7 天才第一次形成完整的 7 天窗口。

所以第三步就很自然了：只保留窗口长度正好等于 7 的结果。

```sql
WHERE day_count = 7
```

这样前 6 天就会被过滤掉，只留下真正满足题意的“完整 7 天统计”。

最后再计算平均值：

```sql
ROUND(amount / 7, 2) AS average_amount
```

因为这里的 `amount` 已经是连续 7 天的总营业额，所以直接除以 `7` 就能得到平均值，再保留两位小数即可。

结合示例来看，`2019-01-07` 这一行对应的是第一段完整的 7 天窗口：

- `2019-01-01 ~ 2019-01-07`

总营业额是：

- `100 + 110 + 120 + 130 + 110 + 140 + 150 = 860`

平均值就是：

- `860 / 7 = 122.86`

所以结果中的第一行就是：

- `2019-01-07, 860, 122.86`

这题本质上考的是**先汇总，再做滑动窗口统计**。  
只要题目里出现“按天统计”“最近 7 天”“移动平均值”“连续几天总和”这类描述，基本就要想到窗口函数，尤其是这种：

`ROWS BETWEEN ... PRECEDING AND CURRENT ROW`

不过在真正上窗口函数之前，一定先确认粒度对不对。  
这题最容易错的地方，不是窗口本身，而是忘了先把同一天的数据合并。
