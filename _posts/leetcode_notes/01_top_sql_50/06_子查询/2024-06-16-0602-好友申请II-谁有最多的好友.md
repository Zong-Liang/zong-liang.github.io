## 题目：0602. 好友申请 II ：谁有最多的好友

**表结构:** `RequestAccepted`

| Column Name  | Type |
| ------------ | ---- |
| requester_id | int  |
| accepter_id  | int  |
| accept_date  | date |

- `(requester_id, accepter_id)` 是这张表的主键。
- 这张表包含发送好友请求的人的 ID，接收好友请求的人的 ID，以及好友请求通过的日期。

**问题描述:**

编写解决方案，找出拥有最多的好友的人和他拥有的好友数目。

生成的测试用例保证拥有最多好友数目的只有 1 个人。

**示例 1:**

**输入:**

`RequestAccepted` 表:

| requester_id | accepter_id | accept_date |
| ------------ | ----------- | ----------- |
| 1            | 2           | 2016/06/03  |
| 1            | 3           | 2016/06/08  |
| 2            | 3           | 2016/06/08  |
| 3            | 4           | 2016/06/09  |

**输出:**

| id  | num |
| --- | --- |
| 3   | 3   |

**解释:**

编号为 3 的人是编号为 1，2 和 4 的人的好友，所以他总共有 3 个好友，比其他人都多。

**进阶:**

在真实世界里，可能会有多个人拥有好友数相同且最多，你能找到所有这些人吗？

## 题解

这题的关键不在“好友申请”，而在于你要先意识到：

**一条通过的好友记录，实际上对应的是两个人各自多了一个好友。**

比如 `1 -> 2` 这条记录表示 1 和 2 已经成为好友。  
虽然表里只存了一行，但在统计好友数时：

- `1` 的好友数要加 1；
- `2` 的好友数也要加 1。

所以这题不能只统计 `requester_id`，也不能只统计 `accepter_id`，而是要把这两列里的用户都算进去。

对应的 SQL 写法如下：

```sql
SELECT id,
       COUNT(*) AS num
FROM (
    SELECT requester_id AS id
    FROM RequestAccepted

    UNION ALL

    SELECT accepter_id AS id
    FROM RequestAccepted
) AS friends
GROUP BY id
ORDER BY num DESC
LIMIT 1;
```

这段查询可以顺着思路分成两步理解。

第一步，先把所有“好友关系中的人”展开成一列：

```sql
SELECT requester_id AS id
FROM RequestAccepted

UNION ALL

SELECT accepter_id AS id
FROM RequestAccepted
```

这里的意思是：

- 每条好友关系里，发起请求的人记一次；
- 接受请求的人也记一次。

这样处理之后，原来一条记录就会变成两条统计信息。  
比如表里有：

- `(1, 2)`
- `(1, 3)`
- `(2, 3)`
- `(3, 4)`

展开之后就会变成：

- `1`
- `1`
- `2`
- `3`
- `2`
- `3`
- `3`
- `4`

这时每出现一次，就表示这个人多了一个好友。

这里一定要用 `UNION ALL`，不能用 `UNION`。  
因为我们要保留所有出现次数，而不是去重。  
如果去重了，好友数量就会被统计错。

第二步，再按用户分组统计出现次数：

```sql
GROUP BY id
ORDER BY num DESC
LIMIT 1
```

这里的逻辑就很直接了：

- `GROUP BY id`：按用户分组；
- `COUNT(*)`：统计每个人一共出现了多少次；
- 这个次数，就是他拥有的好友数；
- 再按好友数从大到小排序，取第一名即可。

结合示例来看，编号 `3` 出现在这些好友关系里：

- `1 -> 3`
- `2 -> 3`
- `3 -> 4`

所以他一共出现了 3 次，说明他有 3 个好友，分别是 `1`、`2` 和 `4`。  
而且题目已经保证最多好友的人只有 1 个，所以最后直接 `LIMIT 1` 就可以得到答案。

这题本质上考的是一个很常见的转换思路：

**表里存的是“关系”，但题目要的是“每个点的次数统计”。**

像这种一条记录同时和两个人有关的题，往往都要先把关系拆开，再统一统计。  
这一步想明白之后，后面的 `GROUP BY + COUNT` 就很自然了。
